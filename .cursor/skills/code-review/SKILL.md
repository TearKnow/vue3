# 代码审查流程

按顺序执行下列步骤。若用户已贴全量 diff 与范围，可省略补充上下文的追问。

## 1. 收集上下文

- 若未提供：根据已打开文件推断，或追问 **范围**（哪些文件/哪次 PR）与 **改动目的**。
- 优先使用 **git diff** 或 PR diff；若有基线分支，注明。

## 2. 审查清单（逐项扫过）

| 维度 | 关注 |
|------|------|
| 正确性 | 逻辑错误、边界、错误分支、异步/竞态、差一错误 |
| API 与契约 | 破坏性变更、prop/emit 类型、对外导出 |
| Vue / UI | 响应式误用、生命周期泄漏、`v-for` 的 key、基础可访问性 |
| 安全 | XSS（`v-html`）、不安全 URL、密钥、用户输入处理 |
| 性能 | 渲染中多余计算、热点处缺缓存、过重依赖 |
| 可维护性 | 命名、重复、文件体量、非显而易见处是否需注释 |
| 测试 | 行为变更是否缺测、易碎测试模式 |

## 3. 严重级别（统一使用）

全程使用：**阻断（合并前必须修）** · **严重** · **一般** · **建议**（nit，可选）。

## 4. 输出格式（始终使用此结构）

```markdown
## 概要

[1～3 句：改了什么、整体风险]

## 阻断项

- [ ] …

## 发现

### 严重

- …

### 一般 / 建议

- …

## 疑问 / 假设

- …

## 建议后续（非阻断）

- …
```

若无阻断项，在「## 阻断项」下写：**无。**

## 5. 语气

直接、具体：尽量标注 **文件 + 符号或行号**；优先给出修改建议或伪代码，避免空泛表述。
