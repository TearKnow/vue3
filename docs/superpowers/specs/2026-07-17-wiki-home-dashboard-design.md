# Wiki 个人工作台设计

**日期**：2026-07-17  
**状态**：已确认（含动态继续学习与计划同步）

## 目标

将 `/wiki` 从面向新用户的操作说明页改为面向站点所有者的个人知识工作台，让首页能够快速回答：

- 最近在学习什么；
- 下一步应该阅读什么；
- 哪些页面最常使用；
- 知识库最近更新了什么；
- 近期还计划整理什么。

## 页面结构

### 1. 顶部概览

- 标题使用“我的知识库”。
- 简介使用“记录学习过程、技术实践与问题总结”。
- 自动展示文档总数、主题总数和最近更新时间。
- 主题数按 Wiki 根目录下的一级目录去重统计。

### 2. 继续学习

- Wiki 详情页成功加载有效文章后，将规范化路径写入 `wiki-last-read` Cookie。
- Cookie 仅保存路径，有效期一年，使用 `SameSite=Lax`；标题、主题和日期始终从当前 Wiki 元数据解析。
- 首页优先匹配最后阅读路径；Cookie 为空或目标文章已不存在时，回退到最近更新的一篇。
- 没有任何可浏览文档时隐藏该区块。
- 提供“继续阅读”主入口，不再维护写死的当前学习配置。

### 3. 常用页面

- 通过首页代码中的轻量配置维护，避免新增管理后台或配置文件。
- 首版使用现有 Wiki 页面作为入口，不展示不存在的占位链接。
- 卡片展示标题与简短说明。

### 4. 最近更新

- 从 `@nuxt/content` 查询结果自动生成。
- 按文档 `date` 从新到旧排列，展示最多 5 篇。
- 每项展示标题、一级目录和日期，并可进入对应页面。
- 缺少日期的文档排在有日期文档之后，不因字段缺失导致首页报错。

### 5. 近期计划

- 数据保存到仓库的 `data/wiki/plans.json`，通过 GitHub Contents API 读取和写入，以支持跨设备同步。
- 首页只展示 `status: active` 的计划。
- 编辑模式支持新增、修改文案、拖动排序和标记完成，不提供永久删除。
- 标记完成后保留原记录，状态改为 `completed`，并由服务端写入北京时间 `completedAt`；该计划随即从首页活动列表隐藏。
- 编辑模式提供折叠的已完成历史，可将计划恢复为 `active`；恢复时清除 `completedAt`。
- 编辑和保存复用现有 Wiki 密码及 24 小时本地密码缓存。

## 近期计划数据格式

```json
{
  "plans": [
    {
      "id": "稳定唯一值",
      "text": "完成数组与链表基础整理",
      "status": "active",
      "createdAt": "2026-07-17T14:00:00+08:00"
    },
    {
      "id": "另一个稳定唯一值",
      "text": "补充双指针笔记",
      "status": "completed",
      "createdAt": "2026-07-16T10:00:00+08:00",
      "completedAt": "2026-07-17T14:30:00+08:00"
    }
  ],
  "updatedAt": "2026-07-17T14:30:00+08:00"
}
```

- 数组顺序即展示顺序；活动计划与已完成计划各自保持相对顺序。
- 新计划由客户端使用 `crypto.randomUUID()` 生成 `id`；`id` 在创建后不可修改，所有时间由服务端按 `Asia/Shanghai` 生成。
- 服务端保存时必须保留已有 ID，拒绝通过缺失记录的方式永久删除历史。

## 数据与交互

- 首页继续复用 `queryContent('/wiki')`，一次获取 `_path`、`title`、`date` 等所需字段。
- 统计数据和最近更新由查询结果计算；继续学习由 Cookie 与查询结果共同确定。
- 新增 `GET /api/wiki/plans` 读取并规范化实时 GitHub JSON；GitHub 文件暂时不可用时回退到仓库内置的 `data/wiki/plans.json`。响应为 `{ plans, updatedAt, version }`，其中 `version` 为 GitHub SHA。
- 新增 `POST /api/wiki/plans`，请求为 `{ password, plans, version }`：校验 Wiki 密码、计划结构、历史完整性和 SHA 版本，再写入 GitHub；响应返回最新 `{ plans, updatedAt, version }`。
- 客户端进入编辑模式后复制当前数据；点击保存时一次提交完整列表，避免每次输入或拖动都产生 GitHub commit。
- 保存成功后以服务端返回数据更新首页；保存失败时保留编辑内容并展示错误，不提前覆盖已展示数据。
- GitHub SHA 冲突时返回冲突提示，要求重新加载最新计划后再编辑，避免覆盖其他设备上的改动。
- 所有站内入口使用 `NuxtLink` 并添加 `no-prefetch`。
- 路由切换继续使用全局 `useNavigationLoading`，首页挂载后调用 `removeNavigationLoadingOverlay()`。
- 没有 Wiki 文档时保留空状态，不渲染无效入口。

## 计划编辑交互

- “近期计划”卡片提供编辑入口；优先使用已缓存的 Wiki 密码，缓存无效时弹出密码输入。
- 编辑态中每个活动计划包含拖动手柄、文本输入和“完成”操作；完成操作先在本地移入历史区，保存成功后持久化。
- 已完成历史默认折叠，展开后可查看完成时间并执行“恢复”。
- 新增计划默认追加到活动列表末尾。
- 取消编辑放弃本次尚未保存的本地修改。
- 密码弹框遵循遮罩 `pointerdown` / `pointerup` 配对关闭规则，不使用单独的 `@click.self`。

## 视觉方向

- 沿用现有 Wiki 的蓝青色变量、圆角、边框和阴影体系。
- 顶部概览作为视觉主区域，下方使用响应式卡片网格。
- 桌面端优先突出“继续学习”和“最近更新”，移动端改为单列。
- hover 仅改变颜色、边框或阴影，不使用位移与缩放。
- 不新增硬编码颜色。

## 范围

本次修改 Wiki 首页、Wiki 详情页，并新增近期计划读写 API、数据文件和必要的共享类型/工具。不新增独立账户系统，不修改文档 front matter。

## 验收标准

- 首页不再展示“浏览目录、快速新建、随时编辑”三张说明卡。
- 页面包含顶部概览、继续学习、常用页面、最近更新和近期计划。
- 文档数、主题数、最近更新时间和最近更新列表根据 Wiki 内容自动变化。
- 打开任意有效 Wiki 文章后，返回首页的“继续学习”指向该文章；失效路径正确回退。
- 近期计划可新增、修改、排序、标记完成和恢复，并可跨设备读取。
- 已完成计划不在首页活动列表显示，但始终保留在 JSON 和历史区中。
- 客户端不能通过正常编辑流程永久删除任何已有计划。
- 写入计划必须通过 Wiki 密码校验，冲突或失败时不丢失当前编辑内容。
- 桌面端与移动端布局均可正常阅读。
- ESLint 对修改文件无新增错误。
